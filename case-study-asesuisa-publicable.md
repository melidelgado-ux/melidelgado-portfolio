# Asesuisa Case Study

## Español

### Asesuisa

Experiencias web responsive para una marca de seguros mas clara, humana y escalable.

### Overview

Asesuisa es una marca de seguros con presencia digital activa. Mi trabajo comenzo con la migracion del sitio legacy a Webflow, proponiendo mejoras sobre el diseño existente, y evoluciono hacia el rediseño integral del sitio.

El proyecto incluye home, paginas institucionales y landings por tipo de seguro y producto. La propuesta busca mejorar la claridad de navegacion, la consistencia visual, la experiencia responsive, la escalabilidad de componentes y la capacidad del sitio para convertir mejor.

### Mi Rol

UX/UI Designer freelance via Elaniin.

Trabaje en el analisis del sitio actual, benchmark, propuestas de rediseño, construccion en Figma, implementacion en Webflow, definicion de design system, tokenizacion, componentes reutilizables, estructura CMS y validaciones con cliente.

### The Challenge

El sitio actual tenia una base funcional, pero presentaba oportunidades claras de mejora:

- La experiencia visual estaba alejada del manual de marca.
- Se usaban ilustraciones 3D que no estaban contempladas en el sistema visual.
- La paleta se apoyaba casi exclusivamente en el azul institucional.
- Habia poca jerarquia visual.
- Algunos elementos funcionaban en desktop, pero se rompian o requerian adaptaciones separadas en mobile.
- Los headers y banners eran muy similares entre secciones.
- La estructura CMS estaba fragmentada en multiples colecciones para paginas, productos y planes.

El rediseño necesitaba acercar la marca a las personas, mostrar un tono mas humano y contenedor, mejorar la navegacion y preparar el sitio para crecer sin perder consistencia.

### Discovery

Antes de comenzar el rediseño, realizamos una etapa de analisis y levantamiento. La estimacion inicial era de dos meses, pero por necesidades operativas se extendio a tres, casi cuatro meses.

Durante esta etapa analizamos el sitio actual, revisamos competidores y relevamos feedback del stakeholder principal. El benchmark nos ayudo a identificar patrones que funcionaban bien en otras experiencias, puntos debiles frente al sitio actual y oportunidades para justificar decisiones de diseño con mayor fundamento.

Tambien tomamos en cuenta comentarios concretos del cliente: que referencias le resultaban utiles, que elementos generaban ruido y que expectativas tenia sobre la evolucion visual y funcional del sitio.

### Designing for Figma and Webflow

El proceso actual parte de Figma, pero esta pensado desde el inicio para Webflow. Antes de proponer una solucion visual, busco referencias, componentes y librerias ya existentes en Webflow para entender que comportamientos, interacciones y animaciones pueden implementarse de forma realista.

Primero defini tokens de color, tipografia y espaciado en Figma. Luego replique esa tokenizacion en Webflow para mantener consistencia entre diseño e implementacion.

Con esa base, diseñe botones, cards, formularios y componentes reutilizables. En algunos casos, ajuste decisiones directamente en Webflow segun las posibilidades reales de la herramienta y luego replique esos cambios en Figma para mantener la unidad visual del sistema.

Cuando la configuracion nativa de Webflow no alcanza, tambien aplico codigo embebido para resolver animaciones, transiciones o comportamientos especificos.

### Product Landings

El sitio se organiza por tipos de producto, como auto, vida, salud y otros seguros. Dentro de cada categoria pueden existir distintos planes, por lo que una de las decisiones clave fue diseñar componentes preparados para crecer.

Para mejorar la orientacion del usuario, cada tipo de producto utiliza un color de apoyo propio, manteniendo el azul y el verde como colores principales de Asesuisa. Esto permite que las landings mantengan una estructura consistente, pero que cada producto sea mas reconocible.

Tambien propuse reemplazar ilustraciones 3D por fotografias reales de personas, alineadas al manual de marca, para construir una experiencia mas cercana y humana.

En Webflow, una parte importante del trabajo fue repensar la estructura CMS. En lugar de sostener multiples colecciones separadas para cada tipo de producto o plan, la propuesta busca consolidar una coleccion mas flexible que pueda servir para distintos planes, reduciendo complejidad y preparando el sitio para futuras iteraciones.

### Outcome So Far

El rediseño continua en construccion, por lo que el impacto final aun debe evaluarse cuando las nuevas secciones esten publicadas. Hasta ahora, el trabajo dejo una base mas solida para evolucionar el sitio:

- Sistema de tokens replicado entre Figma y Webflow.
- Componentes reutilizables para landings y planes.
- Mayor alineacion visual con el manual de marca.
- Navegacion y jerarquia mas claras.
- Propuesta responsive pensada desde el inicio para desktop, tablet y mobile.
- Estructura CMS mas escalable.
- Validaciones semanales con cliente y stakeholders.

Este caso muestra como diseño pensando en la implementacion real: una experiencia no solo debe verse bien en Figma, tambien debe poder validarse, construirse, mantenerse y crecer.

## English

### Asesuisa

Responsive web experiences for a clearer, more human, and scalable insurance brand.

### Overview

Asesuisa is an insurance brand with an active digital presence. My work started with migrating the legacy website to Webflow, proposing improvements over the existing design, and later evolved into a full website redesign.

The project includes the home page, institutional pages, and landing pages by insurance and product type. The goal is to improve navigation clarity, visual consistency, responsive behavior, component scalability, and the website's ability to support conversion.

### My Role

Freelance UX/UI Designer through Elaniin.

I worked on current-site analysis, benchmarking, redesign proposals, Figma design, Webflow implementation, design system definition, tokenization, reusable components, CMS structure, and client validation.

### The Challenge

The current website had a functional base, but several clear opportunities for improvement:

- The visual experience was distant from the brand guidelines.
- It used 3D illustrations that were not part of the visual system.
- The palette relied almost exclusively on the institutional blue.
- Visual hierarchy was weak.
- Some elements worked on desktop, but broke or required separate adaptations on mobile.
- Headers and banners were very similar across sections.
- The CMS structure was fragmented into multiple collections for pages, products, and plans.

The redesign needed to bring the brand closer to people, communicate a more human and supportive tone, improve navigation, and prepare the website to grow without losing consistency.

### Discovery

Before starting the redesign, we went through an analysis and discovery stage. The initial estimate was two months, but due to operational needs it extended to three, almost four months.

During this stage, we analyzed the current website, reviewed competitors, and gathered feedback from the main stakeholder. The benchmark helped us identify useful patterns in other experiences, weaknesses compared to the current website, and opportunities to justify design decisions with stronger reasoning.

We also considered specific client feedback: which references felt useful, which elements created friction, and what expectations existed around the visual and functional evolution of the website.

### Designing for Figma and Webflow

The current process starts in Figma, but is planned from the beginning for Webflow. Before proposing a visual solution, I look for existing Webflow references, components, and libraries to understand which behaviors, interactions, and animations can be realistically implemented.

I first defined color, typography, and spacing tokens in Figma. Then I replicated that token system in Webflow to keep design and implementation consistent.

With that foundation, I designed buttons, cards, forms, and reusable components. In some cases, I adjusted decisions directly in Webflow based on the tool's real possibilities, then replicated those changes back in Figma to maintain visual unity across the system.

When Webflow's native configuration is not enough, I also use embedded code to solve specific animations, transitions, or behaviors.

### Product Landings

The website is organized by product types, such as car, life, health, and other insurance categories. Each category can include different plans, so one of the key decisions was to design components prepared to scale.

To improve user orientation, each product type uses its own supporting color, while keeping blue and green as Asesuisa's main brand colors. This allows the landing pages to keep a consistent structure while making each product easier to recognize.

I also proposed replacing 3D illustrations with real photography of people, aligned with brand guidelines, to create a closer and more human experience.

In Webflow, an important part of the work was rethinking the CMS structure. Instead of maintaining multiple separate collections for each product or plan type, the proposal aims to consolidate a more flexible collection that can serve different plans, reducing complexity and preparing the website for future iterations.

### Outcome So Far

The redesign is still in progress, so the final impact will need to be evaluated once the new sections are published. So far, the work has created a stronger foundation to evolve the website:

- Token system replicated between Figma and Webflow.
- Reusable components for landing pages and plans.
- Stronger visual alignment with the brand guidelines.
- Clearer navigation and hierarchy.
- Responsive proposal planned from the beginning for desktop, tablet, and mobile.
- More scalable CMS structure.
- Weekly validations with the client and stakeholders.

This case shows how I design with real implementation in mind: an experience should not only look good in Figma, it should also be validated, built, maintained, and scaled.
