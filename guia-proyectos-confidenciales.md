# Guia Para Elegir y Contar Proyectos Confidenciales

## Idea Principal

No todos los proyectos necesitan aparecer como case study publico completo. Para un portfolio profesional, especialmente con proyectos de gobierno, ERP, fintech o sistemas internos, conviene separar:

- **Portfolio publico:** proyectos que se pueden mostrar con imagenes reales, links publicos o visuales aprobados.
- **Archivo publico breve:** proyectos que pueden mencionarse sin exponer informacion sensible.
- **Material para entrevista:** proyectos que se pueden contar verbalmente o con visuales anonimizados bajo contexto privado.

Esto permite mostrar experiencia sin comprometer confidencialidad ni depender de capturas internas.

## Recomendacion Actual

Como Asesuisa y Alianza Americas son webs publicas, conviene que sean los dos casos visuales fuertes.

Para el tercer destacado, hay tres caminos posibles:

### Opcion 1: Portfolio as a Product

Usarlo como tercer destacado. Es seguro, actual, publico y permite mostrar estrategia, narrativa, UX writing, arquitectura de informacion y diseño web.

Conviene si:

- No queres forzar un proyecto confidencial.
- Queres mostrar proceso actual y pensamiento estrategico.
- Queres que el portfolio se sienta personal y memorable.

### Opcion 2: Caso confidencial anonimo

Elegir un proyecto como Tegra, ABACO, Elaniin AI, Migraciones o Z11, pero contarlo sin nombres, capturas ni datos sensibles.

Ejemplo:

- "ERP logistics discovery for a manufacturing company"
- "Financial loan simulator and backoffice"
- "Government services platform"
- "AI management platform for internal teams"

Conviene si:

- El proceso fue fuerte.
- Podes hablar del problema, rol y decisiones sin mostrar informacion sensible.
- Podes crear diagramas anonimizados, wireframes recreados o mapas conceptuales.

### Opcion 3: Proyecto solo para entrevista

Mencionarlo en el archivo publico, pero dejar el detalle para entrevistas.

Conviene si:

- Hay confidencialidad alta.
- No tenes visuales publicables.
- El valor principal se entiende mejor conversandolo.

## Preguntas Para Evaluar Cada Proyecto

### 1. Seguridad y confidencialidad

- Puedo nombrar al cliente o conviene anonimizarlo?
- El proyecto esta publicado o es interno?
- Puedo mostrar pantallas reales?
- Puedo mostrar pantallas parcialmente tapadas?
- Puedo recrear pantallas genericas sin exponer informacion?
- Hay datos sensibles, procesos internos, usuarios reales o informacion gubernamental?
- Me sentiria comoda explicando este caso en una entrevista?

### 2. Material disponible

- Tengo acceso a Figma, capturas o prototipos?
- Tengo permiso para mostrar algo?
- Puedo mostrar solo flujos, diagramas o componentes?
- Puedo crear visuales anonimizados?
- Puedo usar texto, diagramas y storytelling sin imagenes reales?
- Hay algun artefacto no sensible: user flow, arquitectura, benchmark, blueprint, tabla comparativa?

### 3. Valor del caso

- Que habilidad demuestra este proyecto que Asesuisa y Alianza no muestran?
- Muestra producto, sistemas complejos, discovery, research, data, fintech, gobierno, ERP o mobile?
- Tuve un rol importante en decisiones?
- Puedo explicar el problema con claridad?
- Puedo explicar una decision concreta que mejore la experiencia?
- Hay algun resultado, aprendizaje o impacto aunque no sea numerico?

### 4. Comodidad narrativa

- Me da orgullo contar este proyecto?
- Lo puedo contar sin entrar en conflictos con cliente, empresa o equipo?
- El cierre del proyecto fue sano o dificil?
- Si fue dificil, puedo enfocarlo en aprendizaje sin sonar defensiva?
- Me representa hoy como diseñadora?

### 5. Potencial visual

- Aunque no tenga capturas, puedo crear:
  - Diagramas de flujo?
  - Arquitectura de informacion?
  - Wireframes anonimizados?
  - Blueprints?
  - Componentes recreados?
  - Tablas antes/despues de decisiones?
  - Journey maps?

## Preguntas Especificas Por Proyecto

### Tegra

- Que procesos se mapearon durante discovery?
- Que entregables hiciste: blueprint, flujos, pantallas, documentacion, presentaciones?
- Que aprendiste trabajando con EY Corea?
- En que momentos usaste ingles?
- Que presentaste en Honduras?
- Que iteraciones surgieron en vivo?
- Que parte se puede contar sin revelar procesos internos?
- Podrias mostrar un blueprint anonimizado o recreado?

### ABACO

- Que tipo de usuario usaba el simulador?
- Que problema resolvia el backoffice?
- Que flujo principal diseñaste?
- Que era dificil de entender antes?
- Que decisiones tomaste para simplificar el simulador?
- Hubo validacion con cliente o desarrollo?
- Que podrias recrear: flujo de simulacion, wireframes genericos, estados de formulario?

### Elaniin AI

- Que usuarios o roles tenia la plataforma?
- Que problema resolvia: accesos, equipos, planes, permisos?
- Que pantallas o flujos diseñaste?
- Que referencias usaste, por ejemplo Google Cloud?
- Que decisiones tomaste para organizar informacion compleja?
- Se puede contar como plataforma corporativa anonima de AI?
- Podes mostrar arquitectura o wireframes recreados?

### Migraciones / Gobierno / Z11

- Que perfiles de usuario habia?
- Que tramites o tareas se gestionaban?
- Que problema de trazabilidad, comunicacion o eficiencia se buscaba resolver?
- Que parte hiciste vos: discovery, flujos, UI, dashboard, app mobile, manual?
- Que no se puede mostrar por confidencialidad?
- Se puede mencionar solo en archivo publico y guardar detalle para entrevista?

## Mi Recomendacion Para El Portfolio

### Publico abierto

- Asesuisa como case study completo.
- Alianza Americas como case study visual/web.
- Portfolio as a Product como tercer destacado.
- Archivo de proyectos con tarjetas breves de todos los demas.

### Para entrevistas

Preparar una carpeta privada o presentacion con proyectos confidenciales, anonimizados:

- Tegra como discovery internacional y procesos complejos.
- ABACO como fintech/backoffice.
- Elaniin AI como plataforma corporativa de AI.
- Gobierno/Migraciones/Z11 como sistemas complejos multiusuario.

## Como Mostrar Proyectos Sin Capturas

En vez de pantallas reales, se pueden usar:

- Diagramas de proceso.
- Flujos anonimizados.
- Wireframes recreados.
- Mapas de arquitectura.
- Ejemplos de componentes sin marca.
- Tablas de decisiones: problema, decision, razon.
- Capturas con contenido tapado, si esta permitido.
- Storytelling del proceso con foco en rol, decisiones y aprendizajes.
